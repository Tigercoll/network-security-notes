## 📚 完整学习内容（必读）

### 一、为什么法律是必修课？（技术本身没有善恶，但人有）

在安全圈，有一句话叫：**“技术练得好，牢饭吃到饱”**。
很多天才少年因为好奇心或想证明自己，跨过了法律的边界。
- **真实案例**：某白帽子发现某知名企业漏洞，未授权抓取了 4000 条数据以证明危害，最终因“非法获取计算机信息系统数据罪”获刑。
- **教训**：法律不看你的“出发点”是否高尚，只看你的“行为”是否违法。

---

### 二、深度原理：刑法中的“紧箍咒”

作为渗透测试员，你必须死记硬背以下三条：

#### 2.1 非法侵入计算机信息系统罪 (第 285 条第一款)
- **核心判定**：只要**进去了**就犯罪。
- **特殊对象**：国家事务、国防建设、尖端科学技术领域的系统。
- **老师提醒**：政府网站、军工网站、核心基建系统，**哪怕有漏洞摆在那，你点一下都算犯罪**。

#### 2.2 非法获取/控制计算机信息系统罪 (第 285 条第二款) —— **【最易触发】**
- **行为 A：获取数据**。利用注入下载数据、利用漏洞读取敏感文件。
- **行为 B：非法控制**。上传 Webshell、获取系统管理员权限、建立反弹连接。
- **入罪标准**：
    - 违法所得 5000 元以上。
    - 影响 20 台以上机器。
    - 账号密码类信息 500 组以上。
- **解读**：你以为拖 500 个账号是为了证明漏洞？在法律眼里你已经够格去喝茶了。

#### 2.3 破坏计算机信息系统罪 (第 286 条)
- **行为**：删库（rm -rf）、修改数据、跑挂服务器（DoS）。
- **后果**：造成系统不能正常运行。

---

### 三、渗透实战中的“五大红线”

在甲方授权的项目中，以下行为依然可能是犯罪：

1.  **越权扩面**：合同写了测 `test.com`，你顺手把 `admin.com` 也测了。**（未授权）**
2.  **数据触碰**：证明 SQL 注入能出数据即可，严禁批量 Dump 数据库。**（过度获取）**
3.  **漏洞公开**：在漏洞修复前，私自将细节发到博客、论坛或朋友圈炫耀。**（二次泄露）**
4.  **敲诈勒索**：联系管理员说“不给钱我就把漏洞卖给黑产”。**（刑事重罪）**
5.  **留后门**：测试结束不清理 Webshell。**（非法控制）**

---

### 四、进阶知识：漏洞提交的正确姿势

如果你发现了未授权目标的漏洞，想做个“好人”，该怎么办？

#### 4.1 官方渠道
- **CNVD (国家信息安全漏洞共享平台)**：国家级，权威。
- **CNNVD (国家信息安全漏洞库)**：侧重于软件补丁。

#### 4.2 企业 SRC (Security Response Center)
- 如：腾讯 TSRC、阿里 ASRC。
- **原则**：只测试其公告允许的范围。
- **避坑**：如果你发现一个公司没有 SRC，也没有公开联系方式，**最安全的做法是不测，或通过 CNVD 匿名提交**。

## 🔧 实践任务

### 任务 1：模拟“合规审查”（30 分钟）
**情景**：你是某安全公司的组长。组员汇报了以下四个操作，请判定哪个合法，哪个违法：
1. 为了证明 SQL 注入存在，他下载了数据库中 `admin` 表的前 3 条数据。
2. 他在甲方授权的内网测试中，发现了一个未在授权名单上的备用服务器，并顺手拿到了 Root 权限。
3. 他在 GitHub 上发布了一个自己写的 Exploit，其中硬编码了某大型门户网站的真实路径。
4. 测试结束后，他删除了所有的 Webshell，但保留了一个隐藏的测试账号。

### 任务 2：编写“漏洞提交说明书”（20 分钟）
**目标**：假设你发现了一个知名网站的越权漏洞。
**操作**：请起草一份提交给该网站 SRC 的邮件草稿，重点展示你的专业性（说明你如何发现了漏洞但**点到为止**，没有下载任何数据）。

## 📝 巩固练习

### 练习 1：判断题
**题目**：我在自己的虚拟机里练习扫描，不小心扫到了外网的服务器并发现了一个高危漏洞。由于我是无意的，所以我可以直接联系该网站要奖励。对吗？
**参考答案**：**错误**。未授权扫描本身已构成不合规。如果联系对方要奖励，极其容易被判定为“敲诈勒索”。

### 练习 2：简答题
**题目**：解释渗透测试中的“最小化测试原则”。
**参考答案**：即在能证明漏洞存在的前提下，执行最少量的操作、触碰最少量的数据、产生最少量的系统影响。例如：用 `id` 命令证明命令执行，而不是 `cat /etc/shadow`。
